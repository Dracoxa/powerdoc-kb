# PowerDoc KB Extractor

当任务涉及以下内容时，优先使用本技能：

- 航天器电源系统设计文档抽取
- PDF / DOCX 表格理解与字段落库
- 拓扑结构、功率通道、控制设备、测试验证信息提取
- 将散乱字段整理为稳定的 `power_system` schema
- 为网页知识库展示生成更可读的 JSON

## 目标

输出结果不是“把原文切碎”，而是生成可入库、可检索、可展示的结构化结果：

1. 保留来源可追溯性
2. 尽量按业务对象建模，而不是按关键词堆砌
3. 优先保留表格中的指标、判据、测试矩阵、交付项
4. 标题、术语解释、说明性段落不要混入规则结果

## 推荐流程

1. 先识别文档类型与解析后端
2. 优先抽取结构化表格，再补正文段落
3. 先做 schema 分组，再做摘要生成
4. 页面展示优先使用中文标签和简洁来源缩写

## 抽取优先级

### 1. 表格

优先抽取：

- 轨道参数与环境条件
- 主要电气技术指标
- 太阳电池翼指标
- 蓄电池组指标
- 控制设备要求
- 寿命要求
- 维修性要求
- 功耗要求
- 可靠性
- 测试项目 / 测试矩阵 / 验证方案

要求：

- 表头、子表头、单位、备注分开保存
- 不要把整行都塞进同一个 value
- 同一单元格中的复合信息应拆成多个子指标

### 2. 拓扑结构

不要只做关键词命中，要提取对象关系，至少覆盖：

- 能源源头
- 功率通道数量
- 调节器 / 控制器 / 保护设备
- 储能设备
- 母线范围
- 充放电路径
- 关键控制模式
- 故障隔离或冗余关系

### 3. 规则与判据

只保留能入库的规则：

- 阈值
- 上下限
- 判定条件
- 持续时间
- 工况
- 适用对象

过滤掉：

- 名词解释
- 标题复述
- 纯背景说明
- 与判据无关的宣传式描述

## 目标 schema

抽取结果优先落到这些桶中：

- `functional_requirements`
- `orbit_environment`
- `electrical_indicators`
- `solar_array_requirements`
- `battery_requirements`
- `control_equipment_requirements`
- `lifetime_requirements`
- `maintenance_requirements`
- `mass_constraints`
- `power_budget`
- `reliability_requirements`
- `acceptance_criteria`
- `test_cases`

## 页面展示要求

- 中文优先
- 来源默认显示缩写
- 摘要要短，不能把原文大段粘出来
- 表格优先展示完整字段，不截断关键值
- 白底原生控件观感要压掉，保持深色一体化风格

## 验证标准

修改抽取逻辑后，至少检查：

1. 指标是否完整，不丢单位与条件
2. 复合字段是否被正确拆分
3. 可靠性、重量、功耗、维修性是否进入正确 schema
4. 表格中的测试用例是否能形成稳定矩阵
5. 页面摘要是否仍然出现大段原文污染
